## Installing a plugin

Open Obscura â†’ **Settings â†’ Plugins â†’ Browse community registry**, then enable any plugin from the list. The client fetches `index.yml` from this repository, verifies the SHA-256 of the downloaded `.zip` against the registry, and runs the plugin in its sandboxed runtime.

Some plugins (TVDB, TMDB) require an API key from the upstream service. Obscura prompts for these on first use; the keys are stored in your local Obscura keychain and passed to the plugin's `execute()` call as the `auth` parameter.

For step-by-step screenshots and troubleshooting, see the [Obscura plugin docs](https://pauljoda.github.io/Obscura/docs/plugins/overview).

## Contributing a new plugin

### 1. Scaffold the directory

```
plugins/<your-plugin-id>/
â”śâ”€â”€ manifest.yml
â”śâ”€â”€ <your-plugin-id>.ts          # or .py for Python plugins
â””â”€â”€ tsconfig.json                # TS only â€” copy from plugins/tmdb/
```

### 2. Write the manifest

Mirror an existing plugin (`plugins/tmdb/manifest.yml` is a good template). Required fields:

| Field | Notes |
|---|---|
| `id` | Lowercase, no spaces. Must match the directory name. |
| `name` | Display name shown to users. |
| `version` | Semver. Bump on every change that affects the zip. |
| `runtime` | `typescript` or `python`. |
| `entry` (TS) | `dist/<id>.js`. |
| `script` (Python) | e.g. `["python3", "<id>.py"]`. |
| `auth` | Optional list of credentials Obscura should prompt for. Omit if the upstream API is public. |
| `capabilities` | Map of capability â†’ `true`. Must match the actions handled in `execute()`. |
| `isNsfw` | Set `true` if the plugin can return adult content by default. |

### 3. Implement the plugin contract

**TypeScript** plugins export a default object:

```ts
export default {
  capabilities: { videoByURL: true, /* ... */ },
  async execute(action, input, auth) {
    // dispatch on action, return a normalized result or null
  },
};
```

**Python** plugins read a JSON request from `stdin` (`{ action, input, auth }`) and write a JSON response to `stdout` (`{ ok: true, result } | { ok: false, error }`). See `plugins/musicbrainz/musicbrainz.py` for the canonical reference.

The full action/input/output schemas â€” including the normalized video, folder, series, and audio result shapes â€” are documented in the [plugin guide](https://pauljoda.github.io/Obscura/docs/plugins/overview).

### 4. Build & test locally

From the repo root:

```bash
npm install        # first time only
npm run build
```

`scripts/build-plugins.mjs` walks every entry in `index.yml`, compiles TypeScript with `tsc`, zips the plugin directory (excluding `*.ts`, `tsconfig.json`, `node_modules`, dotfiles), writes `<id>.zip`, and rewrites `index.yml` with the resulting SHA-256 and version.

Smoke-test by `require`-ing the compiled bundle from a Node REPL or a small driver script and exercising each declared action against the live upstream API.

### 5. Register in `index.yml`

Append a new entry. The build script overwrites `sha256` and `version` from your manifest, so a placeholder is fine on the first build.

```yaml
- id: my-plugin
  name: My Plugin
  version: 0.1.0
  date: '2026-04-25'
  path: plugins/my-plugin/my-plugin.zip
  sha256: PLACEHOLDER
  runtime: typescript
  isNsfw: false
  description: One sentence about what this identifies
  author: Your Name
  capabilities:
    videoByURL: true
    # ... must match manifest.yml
```

### 6. Open a pull request

Commit:

- `plugins/<id>/manifest.yml`
- `plugins/<id>/<id>.ts` (or `.py`)
- `plugins/<id>/tsconfig.json` (TS only)
- `plugins/<id>/<id>.zip`
- `index.yml`

Do **not** commit `plugins/<id>/dist/`. Include a short note in the PR describing the upstream API, any rate limits, and which test inputs you used.

## Versioning & releases

Releases are continuous: every merge to `main` is immediately picked up by Obscura clients on their next registry fetch. The build script enforces that the `sha256` in `index.yml` matches the committed `.zip`, so a version bump and a rebuild are all that's needed to ship.

Bump the `version` field in your plugin's `manifest.yml`, run `npm run build`, commit, and merge. Clients re-download when the version changes.
